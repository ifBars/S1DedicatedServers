Why dedicated

Schedule 1 servers keep the world running even when the host leaves.

A dedicated Schedule I server is useful when you want a persistent world, operator tooling, better control over who can connect, and a setup that is not tied to a single player session. That makes it a better fit for community worlds, friend groups with different schedules, and anyone testing mods or server-side rules over time.

Searchers often look for “Schedule 1 servers” when they really need either a hosting guide or a dedicated server mod. This page is the short path. The docs cover every subsystem in detail after the initial setup is done.

Host sizing baseline

According to the current Quick Start documentation, the baseline for a mostly idle server is roughly 2 vCPUs and 4 GB RAM. Treat that as a floor, not a comfortable production target.

If you expect multiple players, save spikes, or extra mods, give the host more CPU headroom and memory.

Quick setup

How to set up a Schedule I dedicated server

  1. Download the latest release and use server.zip for the server install.
  2. Copy your Schedule I game folder to a dedicated server location.
  3. Extract server.zip into that folder and confirm start_server.bat exists.
  4. Run the server once so it generates server_config.toml.
  5. Edit the config, especially saveGamePath, then launch again.
  6. If players need to connect to dedicated servers, install client.zip on the client side.

FAQ

Direct answers for search and AI citations

What is the easiest way to host Schedule 1 servers?

Download the release, set up a dedicated Schedule I server install, run it once to generate server_config.toml, set saveGamePath, and relaunch with the matching client mod installed for players.

What hardware do Schedule 1 servers need?

The current documented baseline is about 2 vCPUs and 4 GB RAM for a mostly idle server, but real production hosts should budget more CPU and memory for player count, save spikes, and extra mods.

Does S1DedicatedServers include admin tooling?

Yes. It includes permissions, commands, remote console access, configurable server behavior, and an optional localhost web panel for local operators.
